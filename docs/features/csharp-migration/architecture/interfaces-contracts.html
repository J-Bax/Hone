Interfaces & Contracts

The 9 port interfaces in Hone.Core that decouple every assembly in the solution.

1 · Seam Diagram — Why Contracts Live in Core

Why do contracts live in Hone.Core?  To prevent circular dependencies. Both consumers (Orchestration, Agents, Lifecycle) and implementers (K6, Git, CopilotCli) reference Hone.Core without knowing about each other. This is the classic Dependency Inversion Principle — high-level policy depends on abstractions, and low-level details depend on the same abstractions. The DI container wires concrete implementations at composition root (Hone.Host).

3 · Cross-Cutting Patterns

Conventions shared by every interface and DTO in Hone.Core/Contracts:

  • CancellationToken on every async method — All interface methods accept a trailing CancellationToken ct parameter. Cooperative cancellation flows from the host down through orchestrators to leaf operations.
  • Immutable records for all DTOs — Every input, output, and option type is a C# positional record. Immutability eliminates shared-state bugs and makes concurrency safe by default.
  • IReadOnlyList / IReadOnlyDictionary for collections — No mutable List<T> or Dictionary<K,V> in public contracts. Consumers cannot accidentally mutate data owned by the caller.
  • Nullable reference types — All optional fields are explicitly T?. The project compiles with <Nullable>enable</Nullable>, so the compiler enforces null-safety.
  • No exceptions for expected failures — Result records carry a bool Success property. Exceptions are reserved for truly unexpected errors (I/O failures, bugs). This keeps error handling explicit and avoids costly exception-driven control flow.
  • async Task throughout — Every method returns Task or Task<T>. There is zero sync-over-async (.Result, .Wait()) anywhere in the contract surface.
